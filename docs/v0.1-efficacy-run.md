## A real finding the run surfaced (and how it was handled — §10 grounding)

The first `alpine-clean` run **FAILED**: grype raised `CVE-2026-40200` (high) on `alpine:3.18`.
Investigation (via the product scan path) showed it is attributed to **musl 1.2.4-r3**, and the
advisory text is *"An issue was discovered in musl libc 0.7.10 through 1.2.6"* — so musl 1.2.4 is
genuinely in range. **This is a TRUE positive: a real CVE disclosed after the fixture was authored,
unfixed in Alpine 3.18.** The engine was correct; the *fixture's* "no current CVEs" premise had rotted.

Resolution: refreshed the FP-control target to a currently-patched tag (`alpine:3.21`, verified 0 high+
on 2026-06-28) and documented it as a point-in-time baseline to refresh when a new base CVE lands — the
same "the world moves" discipline as the threat-intel corpus (§7). The engine was **not** changed:
suppressing a true positive would be a recall regression (§2.4) and a grounding violation (§10).

This is exactly the value of a live efficacy gate: it both validated the recall claim and caught a
stale benchmark before it became a misleading red in CI.
